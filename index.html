한국항공대 소프트웨어 공모전
사용 기술 : HTML, CSS, Javascript, Jquery, Nodejs, Ethereum
1, 프로젝트 소개
- 결식아동들이 간편하게 결제를 할 수 있도록 하고, 어른들이 결식 아동 카드를 남용하지 못하도록 블록체인을 사용하여 감시하도록 한다.
- 결식아동들은 결식아동 카드를 창피해하고 가맹점이 어딘지를 몰라 카드를 사용하지 않는다 -> 웹 ,앱을 통한 결제 및 가맹점 정보 제공으로 해결
- 공무원 또는 어른들이 결식아동 카드를 남용한다 -> 블록체인이 가진 투명성을 통해 감시

2, 역할 및 내용

DB로는 mongoDB를 사용하였고, nodejs에서 mongoose를 사용하여 결제 데이터와 유저 데이터를 저장하도록 하였습니다.

항공 수화물 적재 시뮬레이션 웹 페이지 제작

한국항공대 Planning 연구실
사용 기술 : HTML, CSS, Javascript, Jquery
1, 프로젝트 소개
- 공항의 크레인, 지게차, 운반 차량을 이용하여 수화물을 자동으로 비행기에 적재하는 시스템을 개발한다.
- 2019/4/7~2019/4/9일에 있던 도쿄 AI 엑스포 참여를 목표로 Ai Planning을 이용한 공항 수화물 운반 자동화 프로그램을 만든다.

2, 역할 및 내용
시뮬레이션 페이지의 프론트 엔드 부분을 맡았습니다. Ai Planning 결과가 서버 전달되고 클라이언트는 polling을 하여 결과를 받아옵니다. 결과를 가져오면 이를 화면에 렌더링하는 시스템을 개발했습니다. jquery와 css를 이용하여 지게차 30대와 운반 차량 15대, 크레인 10대의 움직임을 표현하였고, 각 상태를 한 눈에 볼 수 있도록 GUI를 설계했습니다.

3, 문제 해결
Planning 결과를 받아 클라이언트 화면에 렌더링하는 과정에서 일정 시간이 흐르면 브라우저가 멈춰버리는 현상이 발생했었습니다. 해당 문제를 해결하기 위해 사용되었던 jquery의 animate 함수의 내부 로직을 분석하였고, animate 함수 내부의 queue가 포화가 되지 않도록 더블 버퍼링을 구현했습니다. 그러나 사실 animate 함수의 문제가 아니였었고, Planning 결과를 받아오는 polling이 1초에 50번이 넘는 횟수로 진행되고 있었기 때문에 브라우저의 ui thread가 죽어버리는 문제가 발생했었던 것이었습니다. 이후 서버 개발자와의 협의를 통해 polling을 1초에 한 번으로 줄임으로써 문제를 해결하였습니다.

청소기 로봇 제어 프로그램 제작

소프트온넷
사용 기술 : HTML, CSS, Javascript, Jquery, Nodejs, ROS, PDDL

- 학교 수업 중 하나인 '산학 프로젝트'로 기업과 연계하여 학점을 받고 프로젝트를 수행하는 수업이었습니다.

1, 프로젝트 소개
- AI Planning 결과를 바탕으로 로봇의 움직임을 제어하여 원하는 위치에 충돌 없이 움직일 수 있도록 한다.

2, 역할 및 내용
5인 팀의 팀장으로서 시스템 구성과 서버 구현, 로봇 드라이버 구현을 맡았습니다. 서버, 클라이언트 화면 구성은 nodejs, jquery로 만들었습니다. 서버에서는 이전 연구실에 일했던 경험을 토대로 수많은 planning 정보를 화면에 뿌리기 위해서는 polling 방식보다는 socket.io를 이용한 웹 소켓 통신이 좋겠다고 생각하여 웹 소켓 통신으로 구성했습니다. 클라이언트 화면 랜딩 페이지, 8puzzle 시뮬레이션 페이지, 경로 계획 planning 시뮬레이션 페이지, 로봇 위치 설정 페이지를 jquery를 사용하여 간단하게 만들었습니다. 또한 ROS 시스템을 사용하여 로봇 제어를 하였고, rosnodejs라는 api를 사용하여 ROS 시스템과 nodejs 서버간의 연결을 할 수 있도록 하였습니다. 로봇은 라즈베리파이를 연결하여 드라이버 코드를 임베딩 시켰고, 드라이버 시스템과 서버를 ROS를 통해 통신하여 웹 클라이언트가 원하는 위치에 두면 로봇이 자동으로 움직일 수 있도록 만드는 역할을 수행했습니다.

3, 문제 해결
nodejs에서 planning 결과를 받아오고 이를 rosnodejs를 통해 ROS 시스템에 메세지를 보내 로봇에게까지 planning결과를 전달하는데 성공하였습니다. 그러나, 문제는 로봇의 제어 문제였습니다. 90도를 회전하라고해도 89도를 회전하거나 91도를 회전하고, 직진성도 보장되지 않아 결과적으로는 목적지에 도달하지 못했습니다. 가장 큰 문제는 로봇의 구조적인 문제였습니다. 양 바퀴가 다른 모터로 이루어져 있었기 때문에 회전량이 달라 직진성과 회전성이 보장되지 못했던 것이였습니다. 저희는 캘리브레이션을 통해 문제를 어느정도 해결했지만 완전히 문제를 해결할 수 없어 컴퓨터 비전을 통해 문제를 해결했습니다. 각 포인트 마다 표시를 두어, 로봇이 자신이 90도에 제대로 회전했는지 알려주는 것 입니다. 가령 바코드를 바닥에 두고, 바코드가 비전 중심에 갈 때까지 회전해야 90도인 것으로 만드는 것 입니다. 하드웨어적 문제를 소프트웨어적으로 해결할 수 있었던 좋은 경험이었습니다.

자전거 경로 계획 페이지 제작

레이시오
사용 기술 : HTML, CSS, Javascript, React, React Redux, Next.js, AntD, GraphHopper, Firebase, git

- 학교 수업 중 하나인 '산학 프로젝트'로 기업과 연계하여 학점을 받고 프로젝트를 수행하는 수업이었습니다.

2, 역할 및 내용
5인 팀의 팀장이었고, 다른 팀원들은 모두 웹 개발이 모두 처음이어서 전체적인 개발을 총괄하였습니다. 주로 시스템 구성과 자전거 길 찾기 페이지 구현을 중점으로 했습니다. 프론트 엔드는 React를 사용하고 백엔드는 Serverless로 Firebase를 사용하는 방식을 채택했습니다. 주로 React Hooks와 Google 지도 API를 이용하여 자전거 길 찾기 페이지, 랜딩 페이지 구현을 담당했습니다. 로그인 상태나 지도의 상태를 저장하기 위하여, 상태 관리로 Redux를 사용하였습니다. 개발에 있어서는 TDD 기법을 사용하여 Jest 테스트 프레임워크와 enzyme 라이브러리로 컴포넌트별로 unit test를 하였고, 구현되지 않은 REST API는 mock을 만들어 테스팅하였습니다.

Klaytn 블록체인을 활용한 의료 통합 플랫폼

한국항공대 소프트웨어 공모전
사용 기술 : HTML, CSS, Javascript , Nodejs, Jquery, Klaytn
1, 프로젝트 소개
- 각각의 대학 병원마다 있는 진료 카드를 관리하기 너무 귀찮다. 심지어 진료 카드 중에는 결제 기능이 없는 카드가 대다수 이다.
- A 병원에서 CT를 찍은 결과가 있고, B 병원에서 해당 CT 결과를 확인하기 위해서는 환자가 A 병원에서 B병원까지 CT를 가져와야 한다.
- 환자는 자신의 진료 기록을 열람하기 위해서는 복잡한 절차를 걸쳐야 한다.
- 위의 문제를 해결하기 위하여 카카오의 Klaytn 블록체인을 통해, 통합 의료시스템을 만들도록 한다.
- 진료 카드를 한 개만 사용해도 되며, klay를 통한 결제 기능도 가능하도록 한다.
- 하나의 블록체인 안에 환자들의 데이터들이 있음으로 A 병원에서 찍은 CT 사진을 B 병원에서 열람 할 수 있다.
- 환자의 진료 기록을 하나의 토큰화하여 환자에게만 귀속되어 볼 수 있도록 만들자, 또한 쉽게 웹 또는 앱에서 진료 기록을 확인할 수 있도록 한다.

2, 역할 및 내용
해커톤이여서 하루동안 개발하였고, 1인 팀으로 대회에 참가하여 프로젝트 구성, 설계 , 구현 모두를 총괄했습니다. nodejs - express를 이용하여 통합 의료 시스템 서버를 만들었고, jquery와 bootstrap을 이용하여 환자 진료 기록 확인 페이지, 결제 화면, 의사 화면 페이지를 만들었습니다. nodejs서버에서 klaytn 블록체인을 이용하기 위해 Caver와 web3.js를 이용하였고, 이미지와 같은 데이터를 저장하기 위해 IPFS를 사용했습니다. 또한, 환자 진료 기록을 ERC-721 토큰화 하여 환자에게 귀속되도록 만들었습니다.

3, 문제
구현상의 문제보다는 잘못된 비즈니스 모델에 의한 문제가 발생했습니다. 2, 역할 및 내용
인턴으로서, 회사에 사용되고 있던 RabbitMQ Java 코드를 Spring AMQP 코드로 바꾸어 구현하도록 과제를 주었습니다. RabbitMQ와 Spring AMQP에 대한 공식 문서, issue들을 정리하여 에러 핸들링 코드, 테스트 코드, 구현체, 인터페이스 등을 만들었습니다. 나아가 실제 마이크로서비스로 MQ를 도입한 사례를 토대로 Spring 코드를 구현해봄으로써 REST와 MQ, RPC의 차이에 대해서 조사하였습니다.

3, 문제

Spring AMQP는 RabbitTemplate 인터페이스로 각종 RabbitMQ 설정을 할 수 있지만, 시스템 하부적인 기능들을 사용하기에는 한계가 있습니다. 가령 connection error로 인한 재연결 구현은 RabbitMQ Java에서는 쉽게 가능하지만, Spring AMQP의 인터페이스인 RabbitTemplate에서는 사용할 수 없습니다. 저는 이러한 문제를 해결하기 위하여 Spring AMQP의 내부 구현을 분석하고, 필요한 부분을 override 하는 방법을 사용했습니다. 가령 Spring AMQP의 autoconfiguration 부분에서 connection 문제를 해결할 수 있는 코드 부분을 configuration으로 가져와 override 하도록 만든 것 입니다.
